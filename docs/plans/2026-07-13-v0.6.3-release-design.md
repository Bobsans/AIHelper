# v0.6.3 Release Design

- Date: 2026-07-13
- Status: Approved
- Release type: Production patch release
- Tag: `v0.6.3`

## Context

`main` is clean and synchronized with `origin/main` at `82e6da4`. The latest release is the non-draft, non-prerelease `v0.6.2`, published from an annotated tag with Linux x64, Windows x64, and macOS arm64 ZIP assets.

Six commits after `v0.6.2` contain runtime and contract hardening, command and plugin boundary hardening, and the UTF-8 prefix-boundary fix. Although the original request called this a minor release, the explicitly selected version `0.6.3` is a Semantic Versioning patch release.

## Goals

- Publish all changes after `v0.6.2` as production release `v0.6.3`.
- Keep the root crate, runtime/API crates, and four dynamic plugins on one version.
- Preserve the annotated-tag and three-archive release convention.
- Validate cross-platform release packaging before publication.
- Monitor the published release until all expected assets are attached.

## Non-goals

- Introducing new product behavior during release preparation.
- Changing plugin ABI or stable JSON contracts beyond the already merged commits.
- Refactoring workspace version management.
- Publishing crates to crates.io.
- Creating a draft or prerelease.

## Considered Approaches

### 1. Release commit, preflight, annotated tag, published Release

Prepare and validate a dedicated release commit, run the existing release workflow manually as a cross-platform packaging preflight, then create an annotated tag and production GitHub Release. This is selected because the changes since `v0.6.2` include platform-sensitive process and plugin work.

### 2. Release commit and direct publication

Skip the preflight and rely on the workflow triggered by `release.published`. This is faster, but cross-platform packaging failures would be discovered only after the public Release exists.

### 3. Let GitHub create the tag

Create the Release against `main` or a commit SHA without first pushing a local tag. This saves one step but produces a less controlled tag flow and diverges from `v0.6.2`.

## Release Preparation

1. Change `0.6.2` to `0.6.3` in all seven workspace package manifests:
   - root `aihelper`
   - `ah-plugin-api`
   - `ah-runtime`
   - `ah-plugin-github`
   - `ah-plugin-gitlab`
   - `ah-plugin-ollama`
   - `ah-plugin-postgres`
2. Synchronize only the corresponding seven local package entries in `Cargo.lock`.
3. Move the accumulated changes into `CHANGELOG.md` under `## [0.6.3] - 2026-07-13`, keeping `[Unreleased]` empty.
4. Derive GitHub release notes from that changelog section.
5. Verify `ah project version` reports `0.6.3` for all seven packages and that no package metadata remains on `0.6.2`.

## Release Notes Scope

The changelog will summarize user-visible and operational outcomes rather than individual commits:

- bounded command and HTTP output behavior;
- deterministic search and byte-safe Git status handling;
- runtime, plugin contract, command, and task boundary hardening;
- GitHub/GitLab workflow and issue handling robustness;
- the valid UTF-8 sniff-boundary correction;
- documentation and regression coverage associated with those changes.

No breaking change notice is required because released command names, plugin ABI major version, JSON field names, and stable error conventions remain compatible.

## Validation and Preflight

Run locally through `ah run check`:

1. `cargo fmt --all -- --check`
2. `cargo test --workspace --all-targets --locked`
3. `cargo build --locked`
4. `cargo build --release --locked`

After the release commit is pushed, dispatch `.github/workflows/release.yml` on `main` and wait with failure propagation. Inspect its jobs and artifacts. The preflight must successfully produce Linux, Windows, and macOS archives before publication; the workflow's macOS matrix entry is best-effort, so the overall run status alone is insufficient.

## Publication Sequence

1. Commit the version, lockfile, and changelog as `chore: release v0.6.3`.
2. Push the release commit to `origin/main`.
3. Complete and inspect the workflow-dispatch preflight.
4. Create annotated tag `v0.6.3` on the exact release commit.
5. Push the tag to `origin`.
6. Create GitHub Release `v0.6.3` with title `v0.6.3`, changelog-derived notes, `draft=false`, and `prerelease=false`.
7. Wait for the release-triggered workflow with failure propagation.
8. Verify release metadata and these assets:
   - `ah-linux-x64.zip`
   - `ah-windows-x64.zip`
   - `ah-macos-arm64.zip`

## Failure Handling

- Do not tag or publish when local validation fails.
- Do not publish when the preflight lacks any expected platform archive; diagnose and fix first.
- If tag creation or push fails, stop before GitHub Release creation and report the exact state.
- If the published workflow fails, preserve evidence by inspecting jobs, warnings, and logs before proposing any corrective mutation.
- Do not overwrite or recreate an existing `v0.6.3` tag or Release without explicit approval.

## Completion Criteria

- `main` and `origin/main` point to the validated release commit.
- Annotated tag `v0.6.3` points to that same commit.
- GitHub Release `v0.6.3` is published as a production release.
- The release workflow completes and all three ZIP assets are present.
- The working tree is clean and the completed milestone is recorded in basic-memory.
